# 全网蒸馏 · 闭环自愈：鲲鹏泛 SQL 性能调优新范式（MongoDB 篇）

> 借助大模型与 harness 工程，我们打造了一个易上手、证据可追溯的鲲鹏场景泛数据库性能调优 Skill —— **perf-kp-sql**。只需少量连接信息，就能自动完成数据采集、规则校验、知识检索与报告生成，**结论挂规则、规则挂出处、出处挂原文**。该能力聚焦鲲鹏生态数据库性能诊断工程化落地，**不替代人工决策，也不让模型现场空想**；目前已适配 MongoDB，后续将逐步覆盖 MySQL、Redis、GaussDB 等常见引擎。

作为突击队成员，在支撑客户时一个绕不开的场景就是性能调优，鲲鹏场景下数据库全链路调优痛点尤为突出。数据库依托硬件、操作系统、业务负载多层架构运行，单一维度指标排查存在明显局限，无法完整定位根因；又存在专家资源稀缺、资料质量良莠不齐、自动化工具不成熟等现实问题，"人拉肩抗"的传统调优方式急需改造。

---

## 一、痛点：鲲鹏 SQL 调优为什么"自动不起来"

### 1. 指标链路长

数据库性能问题既可能来自数据库自身配置，也可能来自硬件、OS、业务，甚至外部进程。随着软件规模扩大，在软硬件全栈调优场景下，要想对繁多的指标进行全面采集、分析，已经远超出人工调优能力范围。

### 2. 工具自动化程度低

现网数据库运维场景中，性能优化整体自动化程度普遍偏低，传统运维工具功能分散、能力有限，关键的**数据采集、关联分析、异常甄别、根因研判**等环节，仍高度依赖人工介入与手动操作。工具操作流程零散、配置复杂，对人员数据库专业储备、底层系统认知和排障经验要求较高，存在明显使用门槛，难以适配规模化、高效率的鲲鹏全栈数据库运维需求。

### 3. 优质资料难获取

官方文档、鲲鹏调优手册、社区帖子、历史问题单都可能有价值，但质量和适用边界参差不齐。在真实的性能调优中，通过搜索引擎往往只能获得通用场景的调优建议；更深层次的性能问题要么找不到对应经验，要么调优手段不可用。

直接与大模型对话同样存在深坑：一是**答非所问**，二是**幻觉**。当大模型找不到完美匹配的资料时，可能会自己编造答案，"看起来合理，但无法落到生产环境"。

### 4. 强依赖专家，经验难以固化

性能专家只有那么几个，但随着鲲鹏生态的发展，对性能诊断与调优有诉求的局点呈指数级增加。依赖专家"人拉肩抗"的方法，不仅时效性上很难满足局点需求，专家也会疲惫，对专家本身也是很大的考验。

## 二、ohsql-plugin：marketplace 上的两个 plugin

`ohsql-plugin` 是一个开源的 plugin marketplace，目前已经收纳了 `cpu-flamegraph` 和 `perf-kp-sql` 两个 plugin。**同一份 `SKILL.md` 在 Claude Code CLI、OpenAI Codex CLI 上都能直接安装与运行**——一次开发，多端可用。

### 1. cpu-flamegraph

一行命令传主机、用户名、进程名，skill 就在远端跑 `perf record` 取样、把火焰图和 Top-N 热点函数（按 self 占比排序）回传到终端展示，支持 **on-CPU** 和 **off-CPU** 两种采样模式。

### 2. perf-kp-sql

面向鲲鹏场景的数据库性能诊断 skill。自动采集 OS / 平台 / 数据库运行态指标，融合 MongoDB + 鲲鹏诊断经验，结合本地知识库给出可追溯的诊断结果与建议，整理成 **CLI + HTML** 两种报告交付。报告输出后还支持自然语言追问，回答会强制走知识库检索 + 引用挂载。

下一章把它的五段闭环拆开讲。

## 三、perf-kp-sql 解剖：从 PLAN 到追问的五段闭环

```
PLAN → 采集 → 诊断 → 报告 → 追问(可选)
```

整条流程模拟了一名性能专家解题时的内心 loop：**先弄清楚环境 → 一次性把现场抓全 → 对照经验找命中 → 把结论挂在证据上交出去 → 留下入口让人继续追问**。下面逐段拆——每段都讲清**你能拿到什么、什么动作不会发生**。

### 3.1 PLAN：发出第一条远端命令前先把家底摸清

PLAN 阶段做四件事，全部完成才会发出第一条远端命令。这一段的存在价值就是：**别拿不全参数就盲跑、别采到一半才发现 mongod 没起、别 perf 没装却试图取火焰图**。

1. **历史链接复用**。如果 slash 命令里没传 host，也支持读历史连接列表（最近 5 条），让用户选一条已用过的，或选"新连接"手动输。即使只有 1 条历史也要让用户显式确认——不替用户做决定。
2. **环境建连校验**。必填：`host` / `user` / (`password` 或 `privateKeyPath`)。可选：`port`、`engine`（目前只支持 mongo，默认即 mongo）、`mongo_user` / `mongo_password` / `auth_db`。任何必填字段缺失或格式不合法都会要求补全。
3. **DB 凭据预询问**。数据库凭据不足时主动要求用户补全；如果用户选择不输入，也支持环境实例自动探测。
4. **实例确定 + 环境探测**。一次 SSH 收集三类信号：远端 `mongod` 实例的 pid / bind / port，是否装了 `perf`（决定能不能采 on-CPU 火焰图），是否装了 `offcputime-bpfcc`（决定能不能采 off-CPU 火焰图）。

根据远端火焰图工具完备情况，火焰图能力分为 **oncpu+offcpu / oncpu / none** 三档。**第四步之后不会再发任何探测性 SSH**，防止短时间多次握手触发远端 rate limit / 账号锁定。

### 3.2 采集：一次 SSH 拿完所有现场

采集是整套 skill 体系中**唯一会主动发起远端交互的阶段**，后续诊断和报告全部在本地完成。

**(1) 复用单条 SSH 长连接，全量命令多路复用**

perf-kp-sql 不需要在远端预装任何 Agent，全量观测指标都走本地 OpenSSH。同一目标主机的多次 SSH 请求统一复用 OpenSSH 自带的 **ControlMaster** 机制，共享已认证的 TCP 长连接。服务端只记录单一连接行为，多轮采集都在 SSH 协议内部完成多路复用，**完全不受 PAM faillock、fail2ban、sshd MaxStartups 等连接层限速策略影响**——避免生产环境上"采几下就被锁号"。

**(2) 50 项 OS / 硬件基础指标（phase 1）**

通过单条批量聚合命令，一次性完成多维度基础数据采集：

- 操作系统内核参数（THP、swappiness、dirty_ratio）
- 网络核心配置（somaxconn、TCP 保活参数、连接队列 backlog）
- 磁盘负载状态（iostat 统计、IO 调度器类型、磁盘利用率）
- CPU 底层信息（LSE 指令支持、主频调度策略、NUMA 架构、SMT 开关、BIOS 配置）
- 服务器硬件平台（系统架构、内核版本、虚拟化类型、设备厂商、硬件型号）
- 整机内存容量

单次 SSH 会话内完成全量采集，所有原始数据落地保存为本地 txt 文件。

**(3) 火焰图专项采集（phase 1 子流程）**

仅当 PLAN 阶段探测校验 `perf` 可用时才触发。工具齐全时，同步采 oncpu + offcpu 各 3 秒；仅支持单类时，只采 oncpu 3 秒。原始 SVG 留在远端 `/tmp/cpu-flamegraph_<ts>/` 目录，**只拉 Top-N 热点函数和元数据 JSON 回本地**——大体积 SVG 不进对话上下文，要看完整图直接打开本地文件。

**静默降级**：双采异常自动降级为只采 oncpu，oncpu 也失败则跳过火焰图模块，**全程不阻断主流程**。

**(4) 18 项 MongoDB 运行时核心指标（phase 2）**

依托 PLAN 阶段确认的绑定地址、端口、凭据，通过 `mongosh` 批量采集 WiredTiger 缓存、并发控制 ticket、服务连接池、全局锁状态、系统断言日志、慢操作记录等实时运行指标。

如果遇到连接异常（认证失败、地址端口不匹配等）：**二次反向询问数据库凭据 → 最多 3 轮自动重试 → 失败则跳过本环节**。phase 2 仍按正常完成处理，仅在最终报告里标注"数据库运行时数据缺失"，**不影响规则诊断继续执行**。

**(5) 写入历史记录 + 安全回收 SSH**

采集成功后静默写入连接历史，后续可复用。作业末尾主动 `session-close` 销毁 ControlMaster 套接字；即使忘了关，10 分钟 `ControlPersist` 超时也会自动断开，**避免孤儿连接残留**。

### 3.3 诊断：规则跑判定，知识库挂解释

**(1) 规则引擎批量输出诊断结论（phase 3）**

诊断引擎在本地一次性比对采集结果与 **44 条经过生产校验的基线规则**。每条规则都做了标准化定义：适配引擎范围、指标数据来源、判断条件、风险严重等级、优化建议、官方参考链接——**这些字段都会原样落在最终报告里**。客户回头问"你这个结论是怎么来的"，你可以直接甩出"规则 ID + 命中条件 + 现场实测值 + 官方文档链接"四件套，不用再回去翻文档现编。命中风险按严重程度分层汇总：

| 等级 | 数量 |
| --- | --- |
| Critical | 5 项 |
| Warning | 26 项 |
| Info | 13 项 |

**这一步固定走规则判定而不是 LLM 推理**。性能排查里，大模型临场主观判断极易出现"看起来合理、但落不到生产"的问题。把所有判定逻辑固化成标准化规则，可以做到完整链路溯源——规则 ID、命中条件、设备实际值、官方文档来源一一对应。

模型只看一份 2KB 轻量摘要 JSON；完整诊断结果落本地文件、**不写入对话上下文**——既规避上下文过载，也方便你之后用脚本读 JSON 做二次加工或归档进自己的工单系统。

**(2) 本地知识库智能检索匹配（phase 4）**

规则引擎明确问题现象与风险等级后，再用本地知识库补充根因解析与落地方案。知识库收录 **54 篇权威文档**，覆盖 MongoDB 官方手册、WiredTiger 底层机制说明、Arm 架构、鲲鹏硬件、欧拉系统专项调优规范，蒸馏为轻量化可检索片段。

检索采用**混合匹配架构**：

- **FTS5 三元组**：精准术语检索
- **sqlite-vec 384 维向量**：语义模糊匹配

报告生成阶段，所有 critical / warning 等级问题都会自动挂上 Top3 高关联知识片段。**遇到一个严重项时不用再额外去搜索引擎里捞它的根因解释**——相关权威资料就在条目下方，且每段都标了来源链接，你可以直接顺着读下去。

> 诊断全流程**禁止任何远端 SSH 请求**。所有工具探测都已在 PLAN 阶段一次性落地，进入本地诊断后不再连远端。

### 3.4 报告生成：终端读得快，HTML 给得出

诊断结束你**同时**拿到两份产物，给两类场景用：

**(1) 终端摘要**：还在 SSH 会话里就能直接读完——多模块诊断矩阵（哪几个维度有问题）、各严重等级数量、Top Issues 一览、火焰图 Top-N 热点函数、完整报告本地路径。**判断"接下来要不要继续深挖、补采什么、还是直接定优化项"用这一份就够了**。不同 CLI 自动适配格式（Claude Code 走 Markdown 表格、Codex CLI 走 ASCII 边框），中英文宽度对齐已经在前端层算好，**LLM 不参与表格重排，不会出现错位**。

**(2) HTML 报告**：整合所有诊断结论、火焰图精简数据、全量采集文件路径，自动生成独立 HTML。**可以直接发给客户、贴进工单、附在故障复盘文档里**——"评审材料"基本不用再加工。

### 3.5 用户追问：自然语言进，引用挂出处出

诊断跑完 agent **不会主动退出**。你可以直接对着报告追问，例如"WiredTiger 缓存命中率低的底层影响是什么""这个内核参数为什么这样调""优化方案要不要重启服务"——所有非斜杠命令开头的自然对话都会自动走知识库检索，取 Top5 高关联片段整合回复，**每条关键依据都标注 `[参考N]`**，对应文末权威引用链接。意味着**客户问"你这个建议出处在哪"时，你随手就能甩出链接**，不用回去翻文档现找。

**两条硬性边界**：

- 知识库没收录的，会明确告诉你"当前知识库未收录该相关主题"——**你能立刻分辨现在站在权威资料上、还是站在 LLM 联想里**，不至于事后跟客户解释时掉坑。
- 如果一定要补充通用技术常识，会强制标注"**非知识库来源 · 需自行独立验证**"。临场生成的内容不会被包装成"标准化可靠结论"。

涉及操作命令、代码配置、阈值参数类的回复，只有两条合规出路：挂官方参考索引标注来源；或者启用非知识库声明模板加风险提示。**所有答疑都能溯源、能校验**——这是面对客户时唯一能拿得出去的工作方式。

## 四、技术亮点

把这部分单独拎出来。前面几章讲"做了什么"，这一章讲**这些设计具体替你省下了什么麻烦**——给真正在客户局点上干活、面对 SSH 跳板机、要带着证据交差的内核 / 性能定位人员看。

### 4.1 全网蒸馏：每条规则都有出处

> 这是一条**可复用到下一个数据库**的工程流水线，不是 MongoDB 一次性写死的产物。

针对每一种目标数据库，我们都走同一条 pipeline：**全网爬取官方手册 + 架构白皮书 + 鲲鹏 / 欧拉调优规范 + 社区高质量帖子 → 拆解蒸馏成轻量化知识片段 → 一致性校验后沉淀成启用规则**。

MongoDB 当前的成果：

| 阶段 | 数量 |
| --- | --- |
| 权威文档（蒸馏前） | 44 份 |
| 知识片段（蒸馏后） | 2257 条 |
| 启用规则（进入诊断路径） | 59 条 |

下一种引擎（MySQL / Redis / GaussDB）接入时，**这条蒸馏流水线和规则一致性校验框架是可以直接复用的**——这才是这个 skill 长期价值的护城河，而不是当前命中的 5 条 critical 规则。

### 4.2 端到端闭环：把专家解题的 loop 装进 CLI

> 把人类性能专家做一次诊断时脑子里跑的 loop，整段平移进 skill。

一个有经验的专家做一次性能诊断，大概是这条路径：

```
找资料 → 翻历史案例 → 对着案例复刻采集步骤 → 拿到指标对照知识判断 → 定位 → 回头再翻资料解释根因
```

perf-kp-sql 把这条 loop 工程化了：

| 专家 loop 节点 | skill 对应实现 |
| --- | --- |
| 找资料 | 知识库 54 篇文档 + FTS5 / 向量混合检索 |
| 翻历史案例 | PLAN 阶段历史连接复用、诊断产物本地留存 |
| 复刻采集步骤 | 一次 SSH 跑完 50 + 18 + 火焰图 |
| 指标对照知识判断 | 规则引擎 44 条基线规则确定性判定 |
| 定位 + 解释根因 | critical / warning 自动挂 Top3 知识片段 |
| 回头再追问 | 报告后自然语言追问统一走知识库检索 |

闭环不止跑一次——**追问阶段是同一套检索通道**，让"读完报告再问一句"和"主诊断流程"共享同一套证据基础。

### 4.3 三层证据链：规则、指标、知识库都自带出处

让结论建立在规则和证据上，模型不在现场生成判断。**当前实现的四层约束**：

1. **诊断结论来自结构化规则**。规则里明确了适用引擎、指标来源、判断条件、严重程度、建议动作和来源链接；模型只消费规则引擎已给出的诊断结果，不负责猜"这个指标是不是问题"。
2. **启用规则要过一致性检查**：建议内容是否为空、告警和严重问题是否带来源、来源链接和建议是否匹配、规则引用的指标能否从采集结果里找到。**不过关的规则不进诊断路径**。当前保留的 74 条启用规则全部过了这一关。
3. **报告结论带 evidence trail**。看到一个 critical / warning，能一路追到对应指标、当前值、建议和资料来源，不是让你只能相信一段模型生成文本。
4. **覆盖不到的内容不会被包装成确定结论**。范围内给可追溯解释，范围外画明边界，通用常识不当答案用。

这套机制不追求"什么都能答"，目标只有一个：**在已沉淀的知识和规则范围内，给出可复核的稳定结论**。

> 说明：4.1 节里的 "59 条" 是入诊断路径的启用规则口径，本节 "74 条" 是过一致性检查后的总启用规则口径。两者口径不同——后续会在文档里明确分开。

### 4.4 一次 SSH 跑完全量采集：50 + 18 + 火焰图

> 性能采集最怕"多次握手 → 被远端 rate limit 锁号"，这条流水线把所有命令压进一条 SSH 长连接。

- **ControlMaster 多路复用**：服务端只看到一条 TCP 连接，PAM faillock / fail2ban / MaxStartups 这些限速策略都绕开了。
- **批量聚合命令**：50 项 OS / 硬件指标 + 18 项 MongoDB 运行时指标 + Top-N 火焰图，一次会话拿完。
- **静默降级**：火焰图采不到不阻断主流程；mongosh 连不上自动跳过 phase 2 仍然出报告。
- **大文件不进上下文**：SVG 火焰图留在远端 `/tmp`，本地只拿 Top-N JSON，模型只吃 2KB 摘要。

这套设计让"性能采集"这种本来容易出运维事故的动作，变成一个**接近零侵入的远端读操作**。

### 4.5 还有一层潜力：等"会自己修说明书"的 Agent 接入

> 这一条讲**未来可能性**——也是这个 skill 区别于"一次性脚本"的根本所在。

我们正在打磨一个 SQL 原生 Agent，里面带了一个叫 **skill-doctor** 的机制：每次会话结束顺手对一下 `SKILL.md`（写给 agent 看的工作流说明书）跟实际命令行为有没有脱节，如果脱节就在本地修回来——带 evidence、可回滚，**只动工作流文档，不动规则和知识库**。

为什么需要它？工具会改、命令在新版本里换格式、知识库会增条目，说明书跟现实对不上几乎是必然的事；每条字面级漂移都靠人开 PR 修不现实。等这套 Agent 放出来后，perf-kp-sql 装在上面跑会直接吃到这层红利——用得越多、工作流跟现实越对齐，**经验真正可以沉淀**。

需要强调：**规则和知识库的增删改始终走人工评审**——skill-doctor 不改规则，只补工作流。这里说的"自学习"，不是模型在用户机器上自动训练，也不是 LLM 自己改规则。

## 六、Roadmap

## 七、结语

传统的数据库性能调优方式，已经跟不上海内外鲲鹏生态发展对突击队和运维团队提出的需求。基于"**全链路指标采集自动化 + 诊断结论可溯源 + 经验可持续沉淀**"理念的 `perf-kp-sql` 方案，让鲲鹏场景下多数据库引擎的诊断从"人拉肩抗"变成"可复用、可沉淀、可演化"的工程流水线。

它没有承诺自动修一切问题，也没有承诺"零误判"——但它把**专家做诊断时的内心 loop**、**资料蒸馏路径**和**采集 / 诊断的工程纪律**都明确写了下来。随着更多数据库引擎、更多局点案例、更多规则进入这套体系，下一次面对鲲鹏 + 数据库性能问题的人，可以站在前一次的肩膀上往前走，而不必再从零开始。
