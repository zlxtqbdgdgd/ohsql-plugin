# MongoDB 在线调整 cache_size 慢的深度分析

## 问题描述

```javascript
db.adminCommand({
setParameter: 1,
"wiredTigerEngineRuntimeConfig": "cache_size=2GB"
})
```

**现象**：在 mongod 负载高时，这个命令可能会很慢（几秒到几十秒甚至更久）

---

## 核心原因总结

### 快速答案

**在线调整 cache_size 慢的本质原因：**

1. **锁竞争**：需要持有 `reconfig_lock`，与其他重配置操作互斥
2. **Cache 压力**：调小 cache 时触发激进的 eviction，需要等待大量页面逐出
3. **Page 状态等待**：页面可能是脏的、被 pin 的、正在 reconcile 中，无法立即逐出
4. **I/O 瓶颈**：eviction 需要将脏页写入磁盘，受磁盘 I/O 速度限制
5. **Checkpoint 冲突**：可能需要等待 checkpoint 完成，尤其是在处理 History Store 时
6. **应用线程阻塞**：高负载下应用线程可能被迫参与 eviction，影响整体性能

```c
static WT_INLINE bool
__wt_evict_needed(
WT_SESSION_IMPL *session, bool busy, bool readonly, double *pct_fullp)
{
WT_CACHE *cache;
WT_CONNECTION_IMPL *conn;
double dirty_target, dirty_trigger, pct_dirty, pct_full, pct_updates;
double target, trigger, updates_target, updates_trigger;
bool clean_needed, dirty_needed, updates_needed;

conn = S2C(session);
cache = conn->cache;

// 📊 获取各个阈值配置
target = cache->eviction_target; // 默认 80%
trigger = cache->eviction_trigger; // 默认 95%
dirty_target = cache->eviction_dirty_target; // 默认 5%
dirty_trigger = cache->eviction_dirty_trigger; // 默认 20%
updates_target = cache->eviction_updates_target; // 默认 0%
updates_trigger = cache->eviction_updates_trigger; // 默认 0%

// 计算当前使用率
pct_full = __wti_evict_use_pct(session, &pct_dirty, &pct_updates);

// 🔥 判断是否需要 eviction
clean_needed = pct_full >= trigger; // 总使用率 >= 95%

if (!readonly && !busy) {
dirty_needed = pct_dirty >= dirty_trigger; // 脏页率 >= 20%
updates_needed = pct_updates >= updates_trigger;
}

// 计算 cache full 百分比
if (pct_fullp != NULL)
*pct_fullp = WT_MAX(0.0,
100.0 - WT_MIN(
100.0, WT_MAX(clean_needed ? 100.0 * (pct_full - trigger) / (100.0 - trigger) : 0.0,
WT_MAX(dirty_needed ? 100.0 * (pct_dirty - dirty_trigger) / (100.0 - dirty_trigger) :
0.0,
updates_needed ?
100.0 * (pct_updates - updates_trigger) / (100.0 - updates_trigger) :
0.0))));

return (clean_needed || dirty_needed || updates_needed);
}
```

#### 3.3 Cache Stuck 检测

**文件**: `src/evict/evict_inline.h:30-60`

```c
/*
* __wt_evict_cache_stuck --
* 检查 cache 是否"卡住"（eviction 无法跟上压力）
*
* 这表示即使在强制 eviction 的情况下，cache 仍然超过 trigger 阈值
*/
static WT_INLINE bool
__wt_evict_cache_stuck(WT_SESSION_IMPL *session)
{
WT_EVICT *evict;
WT_CONNECTION_IMPL *conn;
uint64_t bytes_inuse, bytes_max, cache_full_threshold;

conn = S2C(session);
evict = conn->evict;

// 只在非内存模式下检测
if (F_ISSET(conn, WT_CONN_IN_MEMORY))
return (false);

bytes_inuse = __wt_cache_bytes_inuse(conn->cache);
bytes_max = conn->cache_size;

// 阈值: cache_size 的 trigger 比例 (默认 95%)
cache_full_threshold =
(bytes_max * conn->cache->eviction_trigger) / 100;

/*
* 如果当前使用量 > 阈值，并且已经尝试了一段时间仍然无法降下来
* 则认为 cache 已经"卡住"
*/
return (bytes_inuse > cache_full_threshold &&
evict->cache_stuck_time != 0 &&
__wt_clock(session) - evict->cache_stuck_time >
conn->cache_stuck_timeout_ms * WT_THOUSAND);
}
```

**调小 cache_size 时的影响**：
```
cache_size 从 8GB → 2GB:

cache_full_threshold = 2GB * 95% = 1.9GB

当前 bytes_inuse = 7GB

判断:
7GB > 1.9GB? → Yes! 🔥

如果持续一段时间 (默认 300 秒) 仍然 > 1.9GB:
→ Cache Stuck 状态
→ WiredTiger 会打印告警日志
→ 性能严重下降
```

#### 4.2 Eviction Review: 为什么页面不能被逐出？

**文件**: `src/evict/evict_page.c:700-876`

```c
/*
* __evict_review --
* 检查页面是否可以被逐出
* 
* 返回错误（如 EBUSY）表示页面当前无法被逐出
*/
static int
__evict_review(WT_SESSION_IMPL *session, WT_REF *ref, uint32_t flags, bool *inmem_splitp)
{
WT_BTREE *btree;
WT_CONNECTION_IMPL *conn;
WT_PAGE *page;
uint32_t refcnt;
bool closing, inmem_split, modified;

conn = S2C(session);
btree = S2BT(session);
page = ref->page;
closing = LF_ISSET(WT_EVICT_CALL_CLOSING);

// 1️⃣ 页面正在被其他线程使用？
refcnt = __wt_atomic_load32(&page->ref_count);
if (refcnt > 1) {
// 页面被 pin 住了，不能逐出
return (__wt_set_return(session, EBUSY));
}

// 2️⃣ 页面是否修改过？
modified = __wt_page_is_modified(page);

// 3️⃣ 如果是干净页面，可以直接逐出
if (!modified)
return (0);

// 4️⃣ 脏页面需要特殊处理

/*
* 🔥 如果正在对 History Store 做 checkpoint，
* 并且当前页面不是 HS 页面，
* 并且 cache 中已经充满了 HS 脏内容，
* 则暂时不要逐出这个页面！
*
* 原因：逐出非 HS 脏页会生成更多 HS 内容，
* 而 checkpoint 期间无法逐出 HS 页面，
* 可能导致 cache 使用量超过限制！
*/
if (conn->txn_global.checkpoint_running_hs && // Checkpoint 正在处理 HS
!WT_IS_HS(btree->dhandle) && // 当前不是 HS 页面
__wti_evict_hs_dirty(session) && // HS 脏内容占比高
__wt_cache_full(session)) { // Cache 已满

WT_STAT_CONN_INCR(session, cache_eviction_blocked_checkpoint_hs);
return (__wt_set_return(session, EBUSY)); // ← 返回 EBUSY!
}

// 5️⃣ 如果当前线程禁用了 reconcile（例如正在写 HS）
if (F_ISSET(session, WT_SESSION_NO_RECONCILE))
return (__wt_set_return(session, EBUSY));

return (0); // 页面可以被逐出
}
```

**为什么会返回 EBUSY？**

```
场景分析: MongoDB 高负载 + 调小 cache_size

1. 高负载情况:
- 大量应用线程正在读写页面
- 页面的 ref_count > 1 (被 pin 住)
- Eviction 线程尝试逐出 → 返回 EBUSY
- 需要等待应用线程释放页面

2. Checkpoint 正在运行:
- checkpoint 正在处理 History Store
- 非 HS 的脏页暂时不能逐出
- 返回 EBUSY
- 需要等待 checkpoint 完成

3. 写冲突:
- 页面正在被另一个线程 reconcile
- 当前线程标记了 WT_SESSION_NO_RECONCILE
- 返回 EBUSY
- 需要等待写操作完成

4. History Store 压力:
- HS 脏内容占比过高
- 为防止 cache 进一步膨胀，阻止非 HS 页面逐出
- 返回 EBUSY
- 需要等待 HS 内容被清理
```

```c
* __evict_reconcile --

// 1. 合并页面中的更新链
// 2. 将旧版本写入 History Store
// 3. 构建磁盘镜像
// 4. 写入 I/O
ret = __wt_reconcile(session, ref,
closing ? WT_REC_CALL_EVICT_CLOSING : WT_REC_EVICT, NULL);

// 可能返回 EBUSY
if (ret == EBUSY) {
// Reconcile 失败，页面暂时无法逐出
return (ret);
}

// ... 其他错误处理 ...

return (ret);
}
```

**Reconcile 的耗时操作**：

```
__wt_reconcile() 的工作流程:

1. 📝 遍历页面的所有 key-value 对
- 检查每个 key 的更新链
- 根据 MVCC 可见性选择合适的版本
- 耗时: O(N) 其中 N = 页面中的 key 数量

2. 📚 处理历史版本
- 将旧版本写入 History Store
- 可能需要分配新的 HS 页面
- 可能触发 HS 页面的 eviction (递归!)
- 耗时: O(M) 其中 M = 历史版本数量

3. 🏗️ 构建磁盘镜像
- 压缩数据 (如果启用了压缩)
- 计算 checksum
- 耗时: 取决于页面大小和压缩算法

4. 💾 写入 I/O
- 将页面镜像写入数据文件
- 可能需要分配新的磁盘块
- 可能需要等待 I/O 完成
- 耗时: 取决于磁盘速度 (10-100ms per page)

总耗时: 可能达到 100+ ms per page
```

```c
WT_TXN_GLOBAL *txn_global;
WT_TXN_SHARED *txn_shared;
double pct_full;

// 某些特权 session 跳过检查
if (F_ISSET(session, WT_SESSION_IGNORE_CACHE_SIZE) ||
FLD_ISSET(session->lock_flags,
WT_SESSION_LOCKED_HANDLE_LIST | WT_SESSION_LOCKED_SCHEMA | 
WT_SESSION_LOCKED_TABLE))
return (0);

// 内存模式不阻塞
if (F_ISSET(S2C(session), WT_CONN_IN_MEMORY))
return (0);

// ... 其他跳过条件 ...

// 🔥 关键检查: cache 是否需要 eviction?
if (!__wt_evict_needed(session, busy, readonly, &pct_full))
return (0);

// 如果需要 eviction
if (pct_fullp != NULL)
*pct_fullp = pct_full;

/*
* 如果 cache 已经很满，但当前线程没有活跃的快照，
* 设置一个快照以避免 reconcile 时丢失更新
*/
btree = S2BT_SAFE(session);
if (btree != NULL && !F_ISSET(btree, WT_BTREE_LOOKASIDE) && 
!readonly) {
txn_global = &S2C(session)->txn_global;
txn_shared = WT_SESSION_TXN_SHARED(session);

if (!F_ISSET(&session->txn, WT_TXN_HAS_SNAPSHOT) &&
(txn_global->current != txn_global->oldest_id ||
__wt_atomic_load64(&txn_shared->pinned_id) == WT_TXN_NONE))
__wt_txn_cursor_op(session);
}

// 🔥🔥🔥 强制应用线程参与 eviction!
// 这会阻塞应用线程，直到 cache 压力降低
return (__wt_cache_eviction_worker(session, busy, readonly, pct_full));
}
```

#### 5.2 应用线程 Eviction Worker

```c
/*
* __wt_cache_eviction_worker --
* 应用线程参与 eviction 工作
*/
int
__wt_cache_eviction_worker(
WT_SESSION_IMPL *session, bool busy, bool readonly, double pct_full)
{
WT_CACHE *cache;
WT_CONNECTION_IMPL *conn;
WT_DECL_RET;
WT_TXN_GLOBAL *txn_global;
WT_TXN_SHARED *txn_shared;
uint64_t initial_progress, max_progress;
uint64_t time_start, time_stop;
bool app_thread_evict_aggressive;

conn = S2C(session);
cache = conn->cache;
txn_global = &conn->txn_global;
txn_shared = WT_SESSION_TXN_SHARED(session);

time_start = __wt_clock(session);

// 🔥 应用线程开始逐出页面
// 每次逐出一个页面，直到 cache 压力降低

for (;;) {
// 尝试逐出一个页面
WT_RET(__evict_page(session, false));

// 检查 cache 是否还需要 eviction
if (!__wt_evict_needed(session, busy, readonly, NULL))
break; // Cache 压力降低，可以返回了

// 检查是否超时
time_stop = __wt_clock(session);
if ((time_stop - time_start) / WT_MILLION > 
conn->cache_max_wait_ms)
break; // 超时退出

// 继续下一轮 eviction
}

time_stop = __wt_clock(session);
// 统计应用线程花费在 eviction 上的时间
WT_STAT_CONN_INCRV(session, application_evict_time, 
(time_stop - time_start) / WT_THOUSAND);

return (ret);
}
```

**应用线程被阻塞的影响**：

```
正常情况 (cache 使用率 txn_global.checkpoint_running_hs && // Checkpoint 正在处理 HS
!WT_IS_HS(btree->dhandle) && // 当前不是 HS 页面
__wti_evict_hs_dirty(session) && // HS 脏内容占比高
__wt_cache_full(session)) { // Cache 已满

// 🔥 阻止非 HS 页面逐出！
WT_STAT_CONN_INCR(session, cache_eviction_blocked_checkpoint_hs);
return (__wt_set_return(session, EBUSY));
}
```

**为什么 Checkpoint 会阻止 Eviction？**

```
Checkpoint 流程:

1. Checkpoint 开始
- 固定一个时间点（snapshot）
- 将 cache 中的脏页写入磁盘

2. 处理 History Store
- Checkpoint 需要遍历 HS 文件
- 在这期间，HS 页面不能被逐出！
- 设置标志: checkpoint_running_hs = true

3. 问题场景:
MongoDB 高负载:
- 大量写操作产生 MVCC 版本
- 旧版本写入 History Store
- HS 脏页占用大量 cache (可能 > 20%)

同时调小 cache_size:
- Cache 使用率 >> 100%
- Eviction 被触发
- 尝试逐出非 HS 页面

冲突:
- 逐出非 HS 脏页 → 生成更多 HS 内容
- 但 HS 页面不能逐出 (checkpoint 正在处理)
- Cache 使用量进一步增加
- 形成"死锁"状态！

4. WiredTiger 的保护机制:
- 检测到这种情况
- 暂时阻止非 HS 页面逐出
- 返回 EBUSY
- 等待 checkpoint 完成

结果:
- Eviction 进度停滞
- 应用线程全部阻塞在 eviction 上
- Reconfigure 命令等待 eviction 完成
- 整个系统响应变慢！
```

#### 6.2 Checkpoint 持续时间

```
Checkpoint 持续时间取决于:

1. 脏页数量:
- 高负载: 可能有 GB 级别的脏页
- 每个脏页需要 reconcile 和写入 I/O
- 耗时: 几秒到几十秒

2. History Store 大小:
- 长事务: HS 可能有大量历史版本
- 需要遍历和清理
- 耗时: 几秒到几分钟

3. 磁盘 I/O 速度:
- HDD: 100-200 MB/s
- SSD: 500-3000 MB/s
- 写入 10GB 数据: 10-100 秒

典型场景:
MongoDB 高负载 + 调小 cache_size:

Checkpoint 间隔: 60 秒 (默认)

如果 reconfigure 命令刚好在 checkpoint 期间触发:
- 需要等待 checkpoint 完成 (可能 30-60 秒)
- 然后才能开始 aggressive eviction
- 再等待 eviction 完成 (可能 10-30 秒)

总等待时间: 40-90 秒！
```

### 7. I/O 瓶颈

#### 7.1 Eviction 的 I/O 操作

```
每个脏页的 Eviction 流程:

1. Reconcile (CPU 密集)
- 合并更新链: 10-50 ms
- 构建磁盘镜像: 10-30 ms
- 压缩: 20-100 ms (如果启用)
- 小计: 40-180 ms

2. 写入 I/O (I/O 密集)
- 页面大小: 通常 32KB (默认)
- 写入延迟:
* HDD: 10-20 ms (寻道 + 旋转)
* SSD: 0.1-1 ms (写放大可能更高)
- 小计: 0.1-20 ms per page

3. 总耗时 per page:
- HDD: 50-200 ms
- SSD: 40-180 ms
```

**调小 cache_size 的 I/O 压力**：

```
场景: cache_size 从 8GB 调到 2GB

需要逐出的数据量: 约 5GB (假设当前使用 7GB)

假设平均页面大小: 32KB

需要逐出的页面数: 5GB / 32KB ≈ 163,840 页面

假设有 8 个 eviction worker 线程:

并行逐出:
- 每个线程处理 163,840 / 8 ≈ 20,480 页面
- 每页平均耗时: 100 ms (SSD) 或 150 ms (HDD)

理论耗时:
SSD: 20,480 * 100ms / 1000 = 2,048 秒 ≈ 34 分钟
HDD: 20,480 * 150ms / 1000 = 3,072 秒 ≈ 51 分钟

但实际上:
- 并行度受磁盘 I/O 队列深度限制
- 应用线程也在写入，竞争 I/O 资源
- Checkpoint 可能在同时运行
- 实际耗时可能是理论值的 2-3 倍！

现实情况:
- 磁盘 I/O 饱和
- Eviction 进度缓慢
- Reconfigure 命令长时间无响应
- 应用延迟增加
```

## 结论

**MongoDB 在线调整 cache_size 慢的根本原因**：

1. ✅ **立即生效** + **延迟清理**的机制不匹配
2. ✅ Cache 使用率瞬间暴涨，触发 aggressive eviction
3. ✅ 页面逐出受多重限制（pin、checkpoint、HS 压力、I/O 瓶颈）
4. ✅ 应用线程被迫参与 eviction，影响正常业务
5. ✅ Reconfigure 持有锁等待 eviction 完成，导致命令阻塞

**最佳实践**：
- 渐进式调整
- 避开高峰期
- 提前规划和监控
- 设置合理的超时时间

**根本解决方案**：
- WiredTiger 层面支持异步 reconfigure
- 智能 eviction 调度
- 更好的 cache shrinking 机制
