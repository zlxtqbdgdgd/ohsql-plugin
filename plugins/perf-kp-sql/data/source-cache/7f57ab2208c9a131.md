MongoDB Performance Regression Benchmarking and the Truth Behind Journaling

September 9, 2024

Author

Vinodh Krishnaswamy

Benchmarks

Database Trends

Insight for DBAs

Share this Post:

I have always been wondering about performance regression when upgrading in MongoDB. From MongoDB v3.6, despite continuous improvement in the MongoDB feature development, the growing feature set has not included much, if anything in the way of performance improvements. At Percona, we always strive to boost the overall performance by looking into the customers’ environment and tuning from both the DB and the OS sides. In this blog post, let me walk you through the performance regression tests between two versions v4.4 and v7.0 (let me tell you why I chose v4.4 later for the test in this blog) & also tuning parameters that can boost the performance in higher(or any) versions. But that comes with a catch too!

Performance tuning

As version 7.0 becomes more mature now, many customers are willing to upgrade to the same. The upgrade path in MongoDB is 4.0->4.2->4.4->5.0->6.0->7.0. So whenever the customer upgrades to the next available higher versions (for example upgrading from v5.0 to v6.0), most of them see the performance dip in their metrics and see an increase in the response time of their app than previous. This is especially common for customers who are upgrading from MongoDB v4.4.x and that is mainly because of how the default behavior of write concern changed from v5.0. Yes, as you know, the default global write concern is changed from 1 to “majority” for the replicaset & sharded cluster environments. See here for more details . This changed how MongoDB behaves for every request to maintain the data integrity between the replicaSet members and I believe it is a much-needed feature improvement for most of the customers too. In such cases, I had to do some tests around write performance for one of my customers and I was astonished to see some unexpected results. Let’s dive into those details below:

Strange behavior

A customer had come with a problem in that the write performance dropped drastically after testing their writes between v4.4 and v7.0 as part of their upgrade. So I did tests around this by creating a write load with a million documents inserted into a collection on my system. I have chosen a single-member replicaSet just to keep the environment simple. In MongoDB v4.4, it took 6 minutes with the default settings, but the same set of tests took 127 minutes in v7.0 which was strange for me. I know there is some 5%-15% performance regression between every version, but this difference was huge. It was again due to the writeConcern settings. Let us see the details below:

Digging for the truth

I wanted to see the reason behind this behavior as I thought a single member could not be affected by the writeConcern. I collected a Flamegraph (see here- https://github.com/brendangregg/FlameGraph ) to see where the CPU resources spend a huge amount of time. Running with the default RW concern values and Flamegraph showed the culprit: the Journal Flusher was causing the long execution. Here is a screenshot of the Flamegraph where JournalFlusher is visible at the bottom left. Conn32 was the connection id that was writing.

The documentation tells the below behavior of this journaling – https://www.mongodb.com/docs/v7.0/reference/write-concern/#j-option :

With j: true , MongoDB returns only after the requested number of members, including the primary, have written to the journal. Previously j: true write concern in a replica set only requires the primary to write to the journal, regardless of the w: <value> write concern.

If journaling is enabled, w: “majority” may imply j: true. The writeConcernMajorityJournalDefault replica set configuration setting determines the behavior. See Acknowledgment Behavior for details.

I understood that writeConcernMajorityJournalDefault has something to do here. I did another test while keeping the majority RWConcern and writeConcernMajorityJournalDefault: false in rs.conf() to acknowledge the write operation with writing on memory only instead of waiting until the journal writing into the disk. Usually, mongod acknowledges the writes when a majority of the members write into the on-disk journal file. This time as expected after the change, a million documents were inserted within 10 minutes. So changing the simple acknowledgment behavior made the difference, but at the cost of journal on-disk writing.

Bug report

I have opened a regression JIRA ticket https://jira.mongodb.org/browse/SERVER-92018 for the same. It was closed by saying that the writes work best with w:1 setting which depicts the v4.4 behavior. However, MongoDB aims to improve the situation from version 8.0 in the future and we’ll have to wait for version 8.0 GA.

From JIRA – https://jira.mongodb.org/browse/SERVER-91298 

Unfortunately, the performance changes are caused by multiple code and configuration changes. It is very difficult to analyze the impact of each single feature considering the long history. 

    In the meantime, we’ve implemented many performance improvements in MongoDB 8.0. We’d recommend you to give it a try and see if it provides same or better performance for your workloads compared to earlier versions.  

Conclusion

The journaling helps to recover the data from the last checkpoint if there is any crash recovery. Here, disabling writeConcernMajorityJournalDefault makes MongoDB acknowledge the write operation after a majority of the voting members have applied the operation in memory itself. However, the journal syncs its buffer with the disk every 100ms ( journal.commitIntervalMs ). So disabling it, in the worst case, could roll back the write operations when there is a crash or restarts in the majority of nodes in the replicaSet. One can always play with those settings and test with the crash scenarios to assess if they can disable the writeConcernMajorityJournalDefault to depict the different behavior.
