## 🎯 根本原因分析

### 问题本质

这是一个**多重bug**：

1. **表面问题**：配置格式错误（使用`[]`而不是`()`）
2. **深层问题**：配置解析失败后，状态管理不一致
3. **致命问题**：后台线程访问未初始化的条件变量导致crash

### Bug发生流程

```
时间轴：
T0: 用户执行: io_capacity=[total=200M] (错误格式)
↓
T1: MongoDB调用WiredTiger重配置
↓
T2: __capacity_config() 尝试解析配置
↓
T3: __wt_config_gets(..., "io_capacity.total", ...) 返回 WT_NOTFOUND
↓
T4: 但代码继续执行，可能访问未初始化的cval
↓
T5: capacity_cond条件变量未正确创建或保持为NULL
↓
T6: MongoDB错误地返回"成功"消息
↓
T7: Eviction后台线程继续运行（不知道配置已损坏）
↓
T8: 页面淘汰触发写入操作
↓
T9: __wt_capacity_throttle() 被调用
↓
T10: __capacity_signal() 尝试发送信号
↓
T11: __wt_cond_signal(conn->capacity_cond)
capacity_cond = NULL 或野指针
↓
T12: 尝试访问 NULL + 0x60 偏移
↓
T13: 💥 Segmentation Fault!
```

### 关键代码位置

#### 问题代码1：未检查返回值
**文件**：`src/conn/conn_capacity.c:39`

```c
// 问题：直接使用WT_RET，不检查NOTFOUND
WT_RET(__wt_config_gets(session, cfg, "io_capacity.total", &cval));

// 如果返回WT_NOTFOUND，cval未被初始化
// 后续访问cval.val会读取垃圾数据
```

#### 问题代码2：未检查NULL指针
**文件**：`src/conn/conn_capacity.c:254-264`

```c
static void
__capacity_signal(WT_SESSION_IMPL *session)
{
// ...
// 问题：直接使用capacity_cond，未检查NULL
```

#### 问题代码3：配置失败后未清理状态
**文件**：`src/conn/conn_capacity.c:184-212`


```c

__wt_cond_signal(session, conn->capacity_cond); // 可能crash

if (conn->capacity_cond == NULL) { // 先检查NULL

__wt_cond_signal(session, conn->capacity_cond); // 安全
}
```

#### 修复3：配置失败时清理状态
```c
// 之前：
WT_RET(__capacity_config(session, cfg));

// 之后：
ret = __capacity_config(session, cfg);
if (ret != 0) {
// 清理所有状态
memset(&conn->capacity, 0, sizeof(conn->capacity));
conn->capacity_cond = NULL;
conn->capacity_session = NULL;
WT_RET(ret);
}
```

## 总结

这个bug涉及**三个层面**的问题：

1. **用户层面**：使用了错误的配置语法
2. **解析层面**：配置解析器未能正确处理错误格式
3. **运行时层面**：后台线程访问了未初始化的数据结构

**立即行动**：
1. 重启MongoDB
2. 使用正确格式：`io_capacity=(total=200M)`
3. 应用代码补丁并重新编译

**关键要点**：
- 使用**圆括号 `()`** 不是方括号 `[]`
- 配置失败会导致系统状态不一致
- 补丁确保错误被正确处理而不是crash
