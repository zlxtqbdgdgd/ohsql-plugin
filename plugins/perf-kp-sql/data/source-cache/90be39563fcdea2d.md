# 为什么已删除的数据仍然存储在 WiredTiger 数据文件中

## 🔴 核心矛盾

### 用户观察

```
X=93, Y=83 组合的 56 个 key：

Z=4 → INSERT_ts ≈ 8, DELETE_ts ≈ 9 (非常早期)
Z=1704869 → INSERT_ts ≈ 3409738, DELETE_ts ≈ 3409739
...
Z=1997571 → INSERT_ts ≈ 3995142, DELETE_ts ≈ 3995143

当前时间戳: ≈ 4,000,000
```

**矛盾**：
- Z=4 的 key 在时间戳 9 时就被删除了
- 现在时间戳已经到了 4,000,000
- **理论上早就应该被清理了，为什么还在文件中？**

---

## 答案：页面级别的 Reconciliation

### 核心原因 1：Reconciliation 是页面级操作

**关键概念**：WiredTiger 不是"逐个 key"清理，而是**整页重写**

```
删除流程：

1. 单个 key 被标记 DELETE (tombstone)
↓
2. 内存中：key 的更新链包含 tombstone
↓ 
3. Checkpoint/Eviction 触发 Reconciliation
↓
4. 整个页面被读取并重写
↓
5. 判断页面中的每个 key 是否"全局可见的删除"
↓
6. 如果页面中有任何 key 不是全局可见：
→ 该页面的所有 key 都会被写入（即使某些 key 很旧）
↓
7. 只有当整个页面的所有 key 都是全局可见删除时：
→ 页面才会被标记为空并回收
```

### 核心原因 2：页面包含不同时间的 key

从输出可以看到：

```
页面信息（包含 Z=4 的页面）:
ref_state: disk
oldest_start: (0, 3405518) → 约 3,405,518
newest_stop: (0, 3998745) → 约 3,998,745

这个页面包含的时间范围：
- 最早的 INSERT: 3,405,518
- 最新的 DELETE: 3,998,745
- 时间跨度: 约 593,227 个时间戳单位
```

**关键发现**：
```
这个页面包含：
- Z=4 的 key (DELETE_ts ≈ 9) ← 非常旧
- 其他更新的 key (DELETE_ts ≈ 3,998,745) ← 比较新

页面级别的时间聚合：
oldest_start: 3,405,518 ← 不是 9！
newest_stop: 3,998,745

为什么 oldest_start 不是 9？
→ 因为这是"页面级聚合"，反映的是页面最后一次重写时的状态
→ 不是这个特定 key 的时间戳
```

**代码位置**：`ex_access.c:111-155`


### 关键问题：oldest_timestamp 推进不够快！

**计算**：
```
当前时间戳: current_ts ≈ 4,000,000

oldest_timestamp = 4,000,000 - 60*100,000
= 4,000,000 - 6,000,000
= -2,000,000 → 实际为 0 或很小的值

问题：oldest_timestamp 几乎没有推进！
```

**原因**：
```c

// 条件：4,000,000 > 6,000,000? 
// → false!
// → oldest_timestamp 根本没有被推进！
}
```

## 真相大白：oldest_timestamp 从未推进！

### ex_access.c 的配置问题

```c
#define SNAPSHOT_WINDOW_SECONDS 60 // 60 秒
#define TS_PER_SECOND 100000 // 代码注释说是这个值

// 计算窗口大小
window = 60 * 100000 = 6,000,000

// 但实际运行时：
// current_ts 只到了约 4,000,000
// 4,000,000 SNAPSHOT_WINDOW_SECONDS * 100000) {
// SNAPSHOT_WINDOW_SECONDS = 60
// 100000 是假设的"每秒的时间戳数"
// 60 * 100000 = 6,000,000

new_oldest_ts = current_ts - SNAPSHOT_WINDOW_SECONDS * 100000;

if (new_oldest_ts > *last_oldest_ts) {
snprintf(ts_string, sizeof(ts_string), 
"oldest_timestamp=%lx", (unsigned long)new_oldest_ts);
ERROR_CHECK(conn->set_timestamp(conn, ts_string));

*last_oldest_ts = new_oldest_ts;
*last_oldest_update_time = current_time;

printf(" [Timestamp Management] Advanced oldest_timestamp to %lu\n", 
new_oldest_ts);
}
}
```

**问题**：
1. 条件 `current_ts > 6,000,000` 在程序前期不会满足
2. 你的程序可能在达到 4,000,000 左右时就停止了或做了 checkpoint
3. 导致 oldest_timestamp 从未被推进
4. 所有 DELETE 的 tombstone 都不是"全局可见"
5. 所以所有数据都被保留！

## 页面粒度的保留机制

### 为什么 Z=4 和 Z=1997571 在同一个文件中？

```
B-tree 的页面组织：

Leaf Page A:
包含 key 范围: [aaaa_1_*, aaaa_50_*]
包含的 Z 值: 4, 123, 456, 1704869, 1997571, ...
时间窗口: [很早, 很晚]

判断删除：
- Z=4 的 DELETE_ts = 9 (很早，应该删除)
- Z=1997571 的 DELETE_ts ≈ 3995142 (很新，不能删除)

Reconciliation 逻辑：
- 遍历整个页面的所有 key
- 只要有任何一个 key 的 tombstone 不是全局可见
- 整个页面都会被保留（包括那些很旧的 key）

结果：
- Z=4 虽然很旧，但和 Z=1997571 在同一页
- Z=1997571 不能删除（仍在窗口内）
- → 整个页面被保留
- → Z=4 也被保留
```

### 代码证据

**文件**: `src/reconcile/rec_row.c:730-1006`

```c
// __wti_rec_row_leaf() - 行叶页 reconciliation

/* For each entry in the page... */
WT_ROW_FOREACH (page, rip, i) { // 遍历页面中的每一行

/* Look for an update. */
WT_ERR(__wti_rec_upd_select(session, r, NULL, rip, vpack, &upd_select));
upd = upd_select.upd;

// 对于每个 key 单独判断
if (upd == NULL && __wt_txn_tw_stop_visible_all(session, twp))
upd = &upd_tombstone; // 这个 key 可以跳过

if (upd->type == WT_UPDATE_TOMBSTONE)
goto leaf_insert; // 跳过这个 key

// 否则写入这个 key
__wti_rec_image_copy(session, r, key);
__wti_rec_image_copy(session, r, val);
}

// 关键：即使某些 key 被跳过，
// 其他不能跳过的 key 仍然会被写入
// 它们共享同一个页面
```

### 页面聚合的时间窗口

```
从输出看到的页面信息：
oldest_start: (0, 3405518) ← 页面中最老的 start_ts
newest_stop: (0, 3998745) ← 页面中最新的 stop_ts

这不是单个 key 的时间窗口，而是：
- 整个页面中所有 key 的时间窗口的聚合
- 代表这个页面覆盖的时间范围

页面包含的 key：
- 有些 key 的 start_ts = 3,405,518 (较旧)
- 有些 key 的 stop_ts = 3,998,745 (很新)
- Z=4 的 key 可能在这个页面中（尽管它的真实 ts 很小）
```

## 核心原因分析

### 原因1：oldest_timestamp 推进不足

**ex_access.c 的配置**：

```c
#define SNAPSHOT_WINDOW_SECONDS 60
// 代码中假设：60 * 100000 = 6,000,000

manage_oldest_timestamp(...) {
if (current_ts > 6,000,000) { // ← 问题在这里！
new_oldest_ts = current_ts - 6,000,000;
set_timestamp("oldest_timestamp=...");
}
}
```

**实际情况**：
```
程序运行到 current_ts ≈ 4,000,000 时做了 checkpoint
4,000,000 0）
→ 不是全局可见
→ 必须保留
```

### 原因2：页面混合了不同时代的 key

```
B-tree 的特性：
- 按 key 的字典序排序，不是按时间排序
- 同一页面可能包含：
* 很早插入的 key (Z=4)
* 很晚插入的 key (Z=1997571)

例如：
Page 1: 包含 X=1-50 的所有 key
├─ X=12, Y=8, Z=4 (很早)
├─ X=12, Y=8, Z=1704869 (较晚)
├─ X=12, Y=8, Z=1997571 (很晚)
└─ X=50, Y=999, Z=xxx (混合)

只要这个页面中有任何一个 key 的 tombstone 不是全局可见，
整个页面都会被保留（包括那些很旧的 key）
```

### 原因3：Checkpoint 的时间点快照

```
Checkpoint 流程：

1. 程序运行到某个点，触发 checkpoint
- current_ts ≈ 4,000,000
- oldest_ts ≈ 0 (从未推进)

2. Checkpoint 以当前的 oldest_ts 为基准
- 判断每个 key 是否可以删除
- stop_ts 0
- → 所有都不能删除

3. 写入所有数据到磁盘
- 即使 Z=4 的 key 已经"删除"很久
- 但相对于 oldest_ts=0，它仍然"不可见"
- → 被保留
```

#### Level 2: 页面级别的 Reconciliation

```
WiredTiger 按页面重写，不是按 key
- 一个页面包含多个 key
- 只要页面中有一个 key 不能删除
- 整个页面都被保留
- 包括那些很旧的 key
```

#### Level 3: B-tree 的空间组织

```
Key 按字典序排列，不是时间序
- Z=4 和 Z=1997571 可能在同一页
- 因为它们的 X_Y 相同或接近
- 时间跨度大，但空间上相邻
```

## 解决方案

### 方案1：修改窗口配置（推荐）

```c
// ex_access.c:138 修改为

#define TS_PER_SECOND 100 // 从 100000 改为 100
// 窗口 = 60 * 100 = 6000

if (current_ts > 6000) { // 更容易满足
new_oldest_ts = current_ts - 6000;
conn->set_timestamp("oldest_timestamp=...");
}
```

### 方案2：定期强制推进

```c
// 每 10000 次操作推进一次
if ((i + 1) % 10000 == 0) {
uint64_t new_oldest = current_ts - 60000; // 固定窗口
if (new_oldest > 0) {
snprintf(ts_string, sizeof(ts_string), 
"oldest_timestamp=%lx", (unsigned long)new_oldest);
conn->set_timestamp(conn, ts_string);
}
}
```

### 方案3：Checkpoint 后立即清理

```c
// checkpoint 后推进 oldest_timestamp
if ((i + 1) % CHECKPOINT_INTERVAL == 0) {
session->checkpoint(session, NULL);

// 推进 oldest_timestamp 到 current_ts - 窗口
uint64_t new_oldest = current_ts > 60000 ? current_ts - 60000 : 0;
snprintf(ts_string, sizeof(ts_string), 
"oldest_timestamp=%lx", (unsigned long)new_oldest);
conn->set_timestamp(conn, ts_string);

// 再次 checkpoint 以触发清理
session->checkpoint(session, NULL);
}
```

## 最终答案

### 问题：为什么这么多已删除数据在文件中？

**答案**：**oldest_timestamp 从未推进，导致所有删除都不是"全局可见"**

**具体原因**：
1. ✅ ex_access.c 的窗口阈值设置太大（6,000,000）
2. ✅ 程序运行的时间戳还没达到这个阈值
3. ✅ oldest_timestamp 保持初始值（0）
4. ✅ 所有 DELETE (stop_ts > 0) 都不满足"全局可见"条件
5. ✅ Reconciliation 时保留所有数据
6. ✅ 页面级操作进一步加剧了保留

**验证**：
- 数据文件有 **301,368 条**记录
- 都是已删除的（都有 stop_ts）
- 但 oldest_timestamp ≈ 0
- 所以全部被保留！

这完美解释了你看到的现象！需要修改 `ex_access.c` 中的窗口配置才能让数据被正常清理。
