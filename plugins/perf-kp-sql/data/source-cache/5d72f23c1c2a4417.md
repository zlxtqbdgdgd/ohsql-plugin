# 磁盘已持久化数据的清理代码路径详解

## 问题场景

```
时间线：

T1: stop_ts stop_ts
- 数据在逻辑上已"过时"
- 但仍在磁盘上

T3: 触发 Reconciliation (checkpoint/eviction)
- 读取磁盘页面
- 重新构建新页面
- 跳过过时的数据
- 🔥 此时从磁盘删除
```

**核心问题**：已持久化的数据何时、如何从磁盘删除？

## 二、关键代码详解

### 2.1 读取磁盘页面的值（包含时间窗口）

**文件**: `src/reconcile/rec_row.c` 
**行号**: 757-768

```c
/* Unpack the on-page value cell. */
// 🔥 从磁盘页面解包值，包括时间窗口信息
__wt_row_leaf_value_cell(session, page, rip, vpack);

/* Look for an update. */
// 查找内存中的更新链
WT_ERR(__wti_rec_upd_select(session, r, NULL, rip, vpack, &upd_select));
upd = upd_select.upd;

/* Take the timestamp from the update or the cell. */
// 决定使用哪个时间戳
if (upd == NULL)
twp = &vpack->tw; // 🔥 使用磁盘上的时间窗口（包含 stop_ts）
else
twp = &upd_select.tw; // 使用内存更新的时间窗口
```

**关键结构**：`vpack->tw` (WT_TIME_WINDOW)

```c
// 时间窗口结构（从磁盘读取）
struct __wt_time_window {
wt_timestamp_t durable_start_ts; // INSERT 的持久化时间戳
wt_timestamp_t start_ts; // INSERT 的提交时间戳
uint64_t start_txn; // INSERT 的事务 ID

wt_timestamp_t durable_stop_ts; // DELETE 的持久化时间戳
wt_timestamp_t stop_ts; // 🔥 DELETE 的提交时间戳（关键！）
uint64_t stop_txn; // DELETE 的事务 ID

};
```

### 2.2 判断磁盘数据是否过时

**文件**: `src/reconcile/rec_row.c` 
**行号**: 770-775

```c
/*
* If we reconcile an on disk key with a globally visible stop time point 
* and there are no new updates for that key, skip writing that key.
* 
* 如果我们 reconcile 一个磁盘上的 key，该 key 有一个全局可见的 stop 时间点，
* 并且没有新的更新，则跳过写入这个 key
*/
if (upd == NULL && __wt_txn_tw_stop_visible_all(session, twp))
upd = &upd_tombstone; // 标记为 tombstone
```

**判断逻辑**：
```c
条件1: upd == NULL
→ 内存中没有新的更新

条件2: __wt_txn_tw_stop_visible_all(session, twp)
→ stop_ts raw != WT_CELL_KEY_OVFL_RM) {
/*
* Keys are part of the name-space, we can't remove them. 
* If an overflow key was deleted without ever having been 
* instantiated, instantiate it now so future searches aren't 
* surprised when it's marked as cleared in the on-disk image.
* 
* Key 是命名空间的一部分，我们不能删除它们。
* 如果一个 overflow key 在未被实例化的情况下被删除，
* 现在实例化它，这样未来的搜索不会感到意外
*/
if (ikey == NULL)
WT_ERR(__wt_row_leaf_key(session, page, rip, tmpkey, true));

WT_ERR(__wt_ovfl_discard_add(session, page, kpack->cell));
}

/* Not creating a key so we can't use last-key as a prefix for a subsequent key. */
// 不创建 key，所以不能将 last-key 用作后续 key 的前缀
lastkey->size = 0;

// 🔥🔥🔥 关键：跳过写入这个 key
break; // 跳出 switch，不会调用 __wti_rec_cell_build_*
```

**后续处理**（继续循环到下一个 key）：

**文件**: `src/reconcile/rec_row.c` 
**行号**: 890-908

```c
// Line 890: leaf_insert 标签
leaf_insert:
/* Write any K/V pairs inserted into the page before this key. */
// 写入在这个 key 之前插入的任何 K/V 对
if ((ins = WT_SKIP_FIRST(WT_ROW_INSERT(page, rip))) != NULL)
WT_ERR(__rec_row_leaf_insert(session, r, ins));
} // WT_ROW_FOREACH 循环结束，继续下一个 key
```

**关键发现**：
- tombstone 类型的 update 不会调用 `__wti_rec_cell_build_*` 函数
- 直接 `break` 并跳到 `leaf_insert` 标签
- 继续处理下一个 key
- **结果：这个 key 不会被写入新的磁盘页面**

### 2.4 释放旧页面的磁盘块

**文件**: `src/reconcile/rec_write.c` 
**函数**: `__rec_write_wrapup()` 
**行号**: 2468-2508

```c
/*
* This page may have previously been reconciled, and that information 
* is now about to be replaced. Make sure it's discarded at some point, 
* and clear the underlying modification information, we're creating a 
* new reality.
* 
* 这个页面可能之前已经被 reconcile 过，这些信息现在即将被替换。
* 确保它在某个时间点被丢弃，并清除底层的修改信息，我们正在创建新的现实
*/
switch (mod->rec_result) {
case 0: 
/*
* The page has never been reconciled before, free the original
* address blocks (if any). The "if any" is for empty trees
* created when a new tree is opened or previously deleted pages
* instantiated in memory.
* 
* 页面之前从未被 reconcile 过，释放原始地址块（如果有）
*
* The exception is root pages are never tracked or free'd, they
* are checkpoints, and must be explicitly dropped.
* 
* 例外：root 页面从不被跟踪或释放，它们是 checkpoint，
* 必须被显式删除
*/
if (__wt_ref_is_root(ref))
break;

// 🔥🔥🔥 释放旧页面的磁盘块
WT_RET(__wt_ref_block_free(session, ref));
break;

case WT_PM_REC_EMPTY: /* Page deleted */
break;

case WT_PM_REC_MULTIBLOCK: /* Multiple blocks */
/*
* Discard the multiple replacement blocks.
* 丢弃多个替换块
*/
WT_RET(__rec_split_discard(session, page));
break;

case WT_PM_REC_REPLACE: /* 1-for-1 page swap */
/*
* Discard the replacement leaf page's blocks.
* 丢弃替换叶页面的块
*
* The exception is root pages are never tracked or free'd, they are
* checkpoints, and must be explicitly dropped.
*/
if (!__wt_ref_is_root(ref))
// 🔥🔥🔥 释放替换页面的磁盘块
WT_RET(__wt_btree_block_free(
session, mod->mod_replace.addr, mod->mod_replace.size));

/* Discard the replacement page's address and disk image. */
// 丢弃替换页面的地址和磁盘镜像
__wt_free(session, mod->mod_replace.addr);
mod->mod_replace.size = 0;
__wt_free(session, mod->mod_disk_image);
break;

default:
return (__wt_illegal_value(session, mod->rec_result));
}

/* Reset the reconciliation state. */
// 重置 reconciliation 状态
mod->rec_result = 0;
```

### 2.5 Block Manager 层的空间回收

**文件**: `src/block/block_addr.c` 
**函数**: `__wt_btree_block_free()`

```c
/*
* __wt_btree_block_free --
* Helper function to free a block from the current tree.
* 帮助函数，从当前树中释放一个块
*/
int
__wt_btree_block_free(WT_SESSION_IMPL *session, const uint8_t *addr, size_t addr_size)
{
WT_BM *bm;
WT_BTREE *btree;

btree = S2BT(session);
bm = btree->bm;

// 🔥 调用 block manager 的 free 方法
// 将块标记为可回收
return (bm->free(bm, session, addr, addr_size));
}
```

**Block Manager 的处理**：

1. **标记块为"可回收"**：
- 块地址被加入 free list
- 不会立即物理删除

2. **下次 Checkpoint 时**：
- 更新 block manager 的元数据
- 确认哪些块真正可以回收

3. **空间重用**：
- 后续写入优先使用 free list 中的块
- 实现磁盘空间回收

## 三、触发时机

### 何时触发 Reconciliation？

| 触发器 | 代码位置 | 说明 |
|--------|----------|------|
| **Checkpoint** | `session->checkpoint()` | 定期触发（默认 60 秒），将所有脏页写入磁盘 |
| **Eviction** | `__wt_evict()` | Cache 压力大时，逐出页面以释放内存 |
| **手动触发** | `db.adminCommand({fsync:1})` | MongoDB 手动触发 checkpoint |
| **Compact** | `session->compact()` | 显式压缩操作 |

### ex_access.c 中的触发

**文件**: `wiredtiger-11.3.1/examples/c/ex_access.c` 
**行号**: 360

```c
// 每 50000 次操作触发一次 checkpoint
if ((i + 1) % CHECKPOINT_INTERVAL == 0) {
printf(" Triggering checkpoint at iteration %llu...\n", 
(unsigned long long)(i + 1));

// 🔥 触发 checkpoint，会执行 reconciliation
ERROR_CHECK(session->checkpoint(session, NULL));

}
```

## 四、数据清理的时间线示例

### 场景：ex_access.c 运行过程

```
初始状态:
oldest_timestamp = 0 (未推进)

操作序列:

Iteration 1:
INSERT key1, ts=1
DELETE key1, ts=2

Iteration 50000: (触发第一次 checkpoint)
Reconciliation:
- 读取内存中的 key1
- stop_ts=2, oldest_ts=0
- 判断: 2 tw (start_ts=1, stop_ts=2)
- upd = NULL (内存中没有新更新)
- twp = &vpack->tw (使用磁盘上的时间戳)
- 判断: __wt_txn_tw_stop_visible_all(session, twp)
→ stop_ts(2) SNAPSHOT_WINDOW_SECONDS * 100000) { // = 6,000,000
new_oldest_ts = current_ts - SNAPSHOT_WINDOW_SECONDS * 100000;
// ...
}
```

**实际情况**：
```
程序运行到 current_ts ≈ 4,000,000
条件: 4,000,000 > 6,000,000? → false
结果: oldest_timestamp 保持为 0
```

**影响**：
```
每次 Checkpoint 时:
- 读取磁盘页面，key 的 stop_ts = 9
- oldest_timestamp = 0
- 判断: 9 tw` | `src/reconcile/rec_row.c` | 766 | 使用磁盘上的 stop_ts |
| **3. 判断是否过时** | `__wt_txn_tw_stop_visible_all()` | `src/reconcile/rec_row.c` | 774 | 🔥 核心判断：stop_ts stop_ts)
2. 触发 reconciliation (checkpoint/eviction)
3. 读取到磁盘页面
4. 判断 stop_ts <= oldest_timestamp
5. 跳过写入新页面
6. 释放旧页面块
```

## 结论

**磁盘数据何时被清理？**

答案：**下次 Reconciliation 时**

具体流程：
1. ✅ oldest_timestamp 推进超过 stop_ts
2. ✅ 触发 checkpoint 或 eviction
3. ✅ Reconciliation 读取磁盘页面（Line 758）
4. ✅ 判断 stop_ts <= oldest_timestamp（Line 774）
5. ✅ 跳过写入到新页面（Line 879）
6. ✅ 写入新页面到磁盘
7. ✅ 释放旧页面的磁盘块（Line 2481）
8. ✅ Block manager 标记空间可回收

**关键代码**：
- 判断：`src/reconcile/rec_row.c:774`
- 跳过：`src/reconcile/rec_row.c:879`
- 释放：`src/reconcile/rec_write.c:2481`

这就是 WiredTiger 清理磁盘上已持久化数据的完整机制！
