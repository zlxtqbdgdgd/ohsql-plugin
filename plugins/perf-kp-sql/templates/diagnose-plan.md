**v0.4.5 关键约束 · 用户连续 3 次 REPL 评审**:
- banner **必须** 在 Step 1.3 用户参数校验通过后**立即**渲染 · 在任何 SshExec / Bash / Read / ToolSearchTool 之前
- banner **不实际连接** · 只输出**用户给的字段** · 不调 SSH 探测 OS / 硬件 / DB 版本

**填充规则**:
- `{user}@{host}:{ssh_port}` 用 SSH 连接串。
  - 用户显式给 `port=N` → 写 `root@host:N`
  - 用户未给 port → 写 `root@host:22 (默认)`
  - 用户说 `port 不清楚` / `不知道` → 写 `root@host:22 (默认 · 端口未提供)`
- `{engine}`:
  - 用户给 `engine=mongo` → 直打该 token
  - 用户未给 → 打 `自动识别`
  - **不**填版本(如 `mongod 7.0.31`)· 版本是采集结果 · 不入 banner
- `{symptom}`:用户原话(如 "数据库卡顿")· **未说就原样打"未提供"**,不要脑补"通用调优审计"等
- 操作系统 / 硬件 / DB pid / bind:port / cpu_model / 内存 等 **均不入 banner** · 全部由 Step 2.6 采集后收尾打出
- 禁止加 emoji · 禁止用 Markdown 标题 `#`

## 🚨 格式红线(最重要)

**只能用 `━` 横线 · 绝对不能用 box drawing**。具体:

- ✅ **正确**(模板里就这样):
  ```
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    kunpeng · 全栈数据库性能诊断计划
    目标        {host}:{port}
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ```

- ❌ **禁用**(哪怕觉得更好看 · **不许画**):
  ```
  ┌────────────────────────┐
  │ kunpeng · 全栈诊断计划 │   ← 禁用!│┌┐└┘├┤┬┴ 任何 box-drawing
  ├────────────────────────┤       都会因 CJK 宽度算不准导致右边界参差
  │ 目标主机  : xxx        │
  └────────────────────────┘
  ```

**原因**:模型无法精确计算 CJK 宽度 · `│` 字符两侧的内容宽度不一致时 · 右竖线必然错位
(实战反馈该错误已出现多次)。统一用 `━` 横线 + 缩进文本 · 不画竖线 · 绝对不会错位。

**诊断范围内的分隔线**用 `─`(U+2500 BOX DRAWINGS LIGHT HORIZONTAL)· 单行 · 不封框。

**顶部 / 底部分隔横线**:上下两条都必须是 62 个 `━` 字符 · 线里不能夹任何中文或其它字符。

**对齐规则**:整个 banner 只有**一个缩进级别**(col 2 是 label 起点 · col 13 是 value 起点)。

**外层 label · col 2 开始 · 后面空格填齐到 col 13**:
- "目标" / "现象" / "模式" / "策略"(2 CJK = 4 格)+ 7 空格 = col 13
- "操作系统"(4 CJK = 8 格)+ 3 空格 = col 13;旧"数据库" / "执行流"行已删
- 执行流一行 3 段 ①-③ · **去掉了"修复建议"** · 因为建议直接在综合报告里呈现 · 不是独立阶段产物(业界 Oracle ADDM / MySQLTuner / pganalyze 都不把 recommendations 拆成独立 stage)
- "预计耗时" / "诊断范围"(4 CJK = 8 格)+ 3 空格 = col 13

**诊断范围子表 · 第一列从 col 13 起 · 第二列从 col 26 起**(wait class 比"类别"宽 · 右侧需让位):
- `诊断范围   wait class   诊断内容`:label + 3 空格 + `wait class`(10 Latin 字符)+ 3 空格 + `诊断内容`
- 表格后续行前面**空 13 格** · 让 `CPU / I/O / 内存 / 并发 / 网络` 也在 col 13
- 5 个类别名后的空格数量(各占 col 13 → col 26 共 13 列):
  - "CPU"(3 Latin = 3 格)+ 10 空格 = col 26
  - "I/O"(3 Latin = 3 格)+ 10 空格 = col 26
  - "内存" / "并发" / "网络"(2 CJK = 4 格)+ 9 空格 = col 26

**验证方法**:所有外层 value(root@host / 自动识别 / ... / 类别 / CPU 瓶颈 / 预计耗时值)
都应该落在 col 13 · 子表二级 value(ARM64 指令集 / 磁盘调度 / ...)都应该落在 col 24。
