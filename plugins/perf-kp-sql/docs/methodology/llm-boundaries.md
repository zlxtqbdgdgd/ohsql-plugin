## 1 · 引言:为什么要谈 LLM 分工边界

### 1.1 领导原话

> "不能有 '反正它出来了' 的黑盒环节。每一步都要解释得清楚:为什么这一步必须 LLM 做 / 失败了怎么办 / 我们怎么知道它没乱编。"

### 1.2 LLM 在 perf-kp-sql 里的双重身份

LLM 既是 **分析能力来源**(否则没法把"鲲鹏 mongo cpu 高"映射到 202 case · 没法读懂 mongosh stdout)· 也是 **幻觉风险来源**(凭训练数据编 URL · 把多个 case 字段拼成新根因 · 把 stdout 数字记错单位)。

**分工边界的核心问题不是"用不用 LLM" · 而是"哪一步该交 / 不该交 LLM"**:

- 有规则可写的环节 → 写规则 · 不交 LLM(LLM 即使做对了也是赌)
- 没规则可写但有真值兜底的环节 → 交 LLM · 但配硬证据校验(URL 字面回查 / lint / 阈值卡)
- 没规则也没真值的环节 → 交 LLM · 必须人工抽样校准(LLM 自评等于没评)
- 数据量大到人工不可行 → LLM 唯一选项 · 但准备降级路径

### 1.3 跟蒸馏工程综述的对接

综述 §1 表里有一句关键预测:

> "你这四步里,**第①、③步现在最容易被 LLM 加速**;**第②、④步反而被 LLM 压力放大**——LLM 自信流畅地输出错误信息会逼你把筛选和验证做得比以往更严。"

perf-kp-sql 是这句话的工程化落地:

- **类比 ①检索**(综述)≈ Phase 0/1/2(信息采集 + 现象路由) → LLM 加速
- **类比 ②筛选**(综述)≈ Phase 4(根因判定 · 案例 是否仍成立) → 必须有硬证据兜底
- **类比 ③综合**(综述)≈ Phase 5(报告生成) → LLM 加速
- **类比 ④验证**(综述)≈ Phase 5.4 lint + 5 标签溯源 → 工程化的"反偏倚清单"

下文先给 4 大原则 · 再用 16+3 介入点表把 perf-kp-sql 实际行为对到这 4 个原则上。

## 2 · 4 大原则(P1-P4)

### P1 · 规则可写 → 不交 LLM

如果一个判定可以写成 `if X then Y` 的硬规则 · **不要交给 LLM 做**(即使 LLM 大多数时候做得对 · 偶尔做错就破坏所有下游引用链)。

| 例 | 规则形式 | 不该用 LLM 是因为 |
|---|---|---|
| Phase 4.1 阈值对比 | `if cache_used >= 95% then trigger` | 数字比较没歧义 · LLM 偶尔会"看错小数位" |
| Phase 0.10 NLM 鉴权状态机 | `installed=true && authenticated=true && notebooks 非空` 才算可用 | 必须看硬证据(`--op check --json` 输出) · 不许 LLM 凭"我记得这台机器没装" 推断 |
| Phase 3.1 SSH 命令拼接 | 命令存在 case 的 `collection_method_quote` 字面里 · 占位符替换走 `--command-file` | 引号 / 变量替换交给 LLM 字符串拼接会被本机 shell 提前展开 |

**反例**(Phase 2.1 路由):语义匹配"鲲鹏 ARM mongo CPU 90%+" 到 `cases/INDEX.md` 的 `cpu-high` 子分类 · **没法写成 if-else 规则**(中文措辞千变万化)· 必须交 LLM。

### P2 · LLM 必有失败模式 → 必须有"硬证据兜底"

LLM 一定会犯错(URL 凭印象写 / 数字记错 / 把多个 case 字段拼成新根因)· 设计时必须为每一种失败模式预留 **硬证据校验通道**:

| 失败模式 | 兜底机制 |
|---|---|
| URL 凭印象写 / 编 URL 凑数 | URL 强制溯源:每个 `[参考N]` 必须 verbatim 来自 Phase 2.3 Read 拿到的 case `source_url` 字段 · 或 NLM 返回的 `references[].source_id`(SKILL.md L1110-1158) |
| 报告原子事实漏挂来源标签 | `format-chat.mjs` 的 5 标签 lint(`scripts/format-chat.mjs` L45-130 `lintReport()`)· 漏挂率 > 0 即 exit 2 强制重写 |
| 把多个 case 字段拼成新根因 | 5 标签强制单源 · 不许 `[OBS+案例]` 联合标签 · 多源拆原子(SKILL.md L1004-1031) |
| chat 输出"自由发挥总结收尾" | format-chat self-check 4 规则(`|---|` 必含 / `|` ≥ 30 / 不许 `────` 独占行 / 不许 ≥ 3 个"字段:值"行 · SKILL.md L1220-1227) |

**没"硬证据兜底" 的环节就是黑盒** · 必须改造或人工兜底(详见 §4)。

### P3 · LLM 评 LLM(LLM-as-Judge)→ 必有人工抽样校准

LLM 给 LLM 输出打分有一个根本缺陷:**模型同源偏盲点**——Sonnet 评 Sonnet 的输出,看不见自己的盲点(用户全局指令明文:"评审者模型必须不同于作者模型")。

跟综述 §6.1 三角验证的"研究者三角"对应:**LLM 不能既当作者又当评审** · 必须有第三方校准。

perf-kp-sql 当前的 P3 缺口:

| 介入点 | LLM-as-Judge 在做什么 | 当前校准 | 应有校准 |
|---|---|---|---|
| 节后 2.1.1 case 正交性判定 | 判 case A 跟 case B 是否同根因 / 重复 / 互斥 | 无 | 5% 人工抽样核对 |
| 节后 2.1.2 case 质量评分 | 给每个 case 打 1-5 分(完整性 / 准确性 / 可执行) | 无 | 评分卡分维度 + 5% 抽样人工评 |

(这两点是节后规划 · 当前主流程不涉及)

### P4 · 大量数据 + 人工不可行 → LLM 是唯一选项 · 但必有降级路径

某些环节数据量大到人工不可行(202 case 两两正交性 = 2 万对; 93 BP 全量 NLM 巡检 = 90+ query)· LLM 是唯一可行选项。这种情况设计时必须准备 **降级路径**:

| 介入点 | 主路径(LLM) | 降级路径 |
|---|---|---|
| Phase 2.1 case 路由 | LLM 语义匹配 user 描述 → INDEX top-K | 6+ 命中时 LLM 简短追问;5 轮仍 > 5 个强制带 5 个进 Phase 3 |
| Phase 4.B BP 全量巡检 | NLM batch 93 BP / 7 min | NLM 不可用 → 退化为 仅案例 单源 · 报告头标 "⚠️ NLM 不可用" |
| Phase 1.1 NLP 抽用户问题关键词 | 自然语言抽现象 / 部署形态线索 | Phase 2 case 路由能再补;描述模糊触发巡检模式(Phase 3.B) |

降级路径必须 **写进 SKILL.md spec** · 不许 LLM 在运行时自由决定加 retry / 加 fallback。

## 4 · 现存"黑盒"风险点 + 改进路线

按领导反馈"不能有 '反正它出来了' 的黑盒环节" · 当前 16 介入点中 · 有 4 个仍存在不同程度的**可解释性风险**(原 3 个黑盒风险点 + 5.3 一句话摘要 0.48.0 新增缺兜底)。每点给候选改进方案 + 工程量预估。

### 4.1 风险点 #1 · "主要问题"一句话摘要(5.3 · 假想增项)

**现状**:Phase 5 报告主体只有 6 列表 + 现场观测段 + 参考段 · **没有**"主要问题一句话"摘要(类似 `Top Issue: WiredTiger eviction 配置不匹配 · 风险 high · 立即调整 dirty_target=3%`)。

**用户场景**:领导扫报告希望 5 秒钟看到结论 · 不愿意逐行看 6 列表。

**风险**:如果加这个一句话 · LLM 自由发挥 → 摘要与正文矛盾(正文写"风险 high" · 摘要写"轻微")· 用户产生误判。

**候选改进方案**:

| 方案 | 工程量 | 兜底原则 |
|---|---|---|
| A · 模板填空 | 0.5 天 | P1:Phase 5 第一行 row 的 `确认的根因` + `风险等级` 字段 verbatim 拼成一句话 · 不让 LLM 自由发挥 |
| B · format-chat lint 加交叉引用规则 | 1 天 | P2:lint 校验"摘要里的根因名 必须在主表第一行出现" · 否则 exit 2 |
| C · LLM 写 + self-check | 0.5 天 | P2:LLM 写完后 self-check"摘要里提到的风险等级 是否等于主表第一行?" · 失败重写 |

**推荐**:A(模板填空)· 工程量小 · 完全消除自由发挥空间 · 跟"chat 输出 stdout 字面复制"是同一种治理思路。

### 4.2 风险点 #2 · Phase 6 用户追问(6.1)

**现状**:Phase 6 是用户拿到报告后追问 · 当前**自由问答** · 仅约束"必须走 案例 Read 或 NLM query · 不许凭训练数据回"。

**风险**:

- LLM 答非所问(用户问"swappiness 改了会不会影响 swap 触发" · LLM 答"swappiness 是什么")
- LLM 把 案例 + NLM 答复"调和"成新结论(综述 §5.5 反例:"它会自信地'调和'实际上互相矛盾的来源")
- 6.1 / 6.2 选路错(用户问"NUMA 拓扑" · 应走 NLM os notebook · LLM 选 案例 路径漏一半信息)

**候选改进方案**:

| 方案 | 工程量 | 兜底原则 |
|---|---|---|
| A · 追问 LLM 输出"决策日志"附录 | 1 天 | P2:每条追问回答末尾必须列"我读了哪个 case_id / 发了哪条 NLM query" · 用户能审计 |
| B · format-chat-q 类似的 lint(检查回答里有没有 [CASE] / [NLM] 标签) | 2 天 | P2:跟 Phase 5 5 标签复用同一套 lint |
| C · 追问超 3 轮强制结束 · 让用户重发 `/perf-kp-sql` 拿新报告 | 0.5 天 | P4:数据量大时降级 |

**推荐**:A(决策日志附录)· 工程量小 · 跟主流程 5 标签思路一致 · 可解释性显著提升。

### 4.3 风险点 #3 · "根因主表为什么选这条 case"的推理链(4.3)

**现状**:Phase 4.A 阶段 3 综合判定表(SKILL.md L887-895)写了"案例 偏离 / NLM 确认 → 进主表" 这种规则 · 但**没暴露具体推理链**(具体 LLM 是怎么从 5 个候选 case 里选了这一条 · 排除了另外 4 条)。

**用户场景**:领导问"为什么选 swappiness 而不选 transparent_hugepage?二者都是 mm 类参数。"

**风险**:

- 如果 LLM 当时的判定基于"我训练数据里见过 swappiness 影响更大" → 黑盒
- 如果 LLM 是基于阈值偏离程度 → 这是规则可写的 · 应该写出来

**候选改进方案**:

| 方案 | 工程量 | 兜底原则 |
|---|---|---|
| A · 报告附录"决策日志" | 1.5 天 | P2:每个根因末尾标"为什么是这个 · 排除了哪几个 case_id · 排除原因(`案例 阶段 threshold 没偏离` / `NLM 否认` 等)" |
| B · 引入"排除原因分类"(类似 PRISMA flow) | 3 天 | P2:对 Phase 2 命中的 5 个 case · Phase 4 必须为每个标"进主表 / 不进 + 原因分类" · 综述 §6.2 PRISMA-Lite 直接对应 |
| C · 在 chat 输出加一个折叠段 `<details>决策日志</details>` | 1 天 | P2:用户默认看不到 · 但需要时能展开审计 |

**推荐**:B(PRISMA-Lite 风格)· 工程量稍大但跟综述 §6.2 直接对接 · 是最具长期价值的可解释性改造。

## 5 · 与综述的对应

### 5.1 §1 "①③ 易加速 / ②④ 反被放大" 对照

| 综述步骤 | perf-kp-sql 对应 phase | LLM 加速 / 放大 |
|---|---|---|
| ① 检索 | Phase 0(env probe)+ Phase 2.1(case 路由) | **加速**:LLM 把"鲲鹏 mongo cpu 90%+"映射到 case_id 比手翻 INDEX 快 100 倍 |
| ② 筛选 | Phase 4.A 阶段 2 NLM 二次确认 + 阶段 3 综合判定 | **放大**:LLM 自信调和矛盾(案例 偏离 + NLM 否认时强行写进表)→ 必须用阶段 3 综合判定表(SKILL.md L887-895)硬卡 |
| ③ 综合 | Phase 5.2 报告渲染 | **加速**:LLM 把多 case + NLM references 写成 6 列表比手写快 |
| ④ 验证 | Phase 5.4 format-chat lint + 5 标签溯源 + URL 字面回查 | **放大**:LLM 自由发挥总结 / 编 URL → 必须用 lint(脚本)+ self-check 4 规则(SKILL.md L1220-1227)硬卡 |

**推论**:perf-kp-sql 的工程化重心 · 跟综述预测一致 · 主要落在 **②④ 防放大** 上。具体表现:

- Phase 4 强约束"双源默认 · 单源是降级"(SKILL.md L755-766)
- Phase 5 强约束 URL 强制溯源 + 5 标签 + lint 三层兜底(SKILL.md L1110-1158)
- format-chat self-check 4 规则把"自由发挥总结" 从执行路径上消除(SKILL.md L1229)

### 5.2 §6.1 三角验证对应

综述 §6.1 列了 4 类三角验证:数据 / 方法 / 理论 / 研究者。perf-kp-sql 当前的双源(案例 + NLM)属于哪一种?

| 三角类型 | perf-kp-sql 是否覆盖 |
|---|---|
| **数据三角**(不同来源说同一件事) | ✅ 覆盖:案例(团队蒸馏快照)+ NLM(Google 检索 + Gemini · 拉最新文档 references)是两个独立检索系统 · 注意 NLM references 必须是真实文档链接 · 不算"A 引 B 引 A" |
| **方法三角**(不同方法证同一结论) | 🟡 半覆盖:案例 是"规则匹配 + 阈值"方法 · NLM 是"语义检索 + 生成"方法 · 但二者都是"读文档" · 不是"实测复现" |
| **理论三角**(不同理论框架解释) | ❌ 未覆盖 |
| **研究者三角**(多人独立得出同一结论) | ❌ 未覆盖(LLM 评 LLM 不算 · 详见 P3) |

**改进建议**:

- **方法三角扩到三源**:在 Phase 3 采集环节增加"对照机器跑同 workload"分支(综述 §8 案例 A 推荐的"自己在测试机复现")· 当前未做 · 工程量较大(需要测试集群)
- **研究者三角**:节后 case 正交性判的 5% 人工抽样 = 一种轻量研究者三角 · 但只覆盖 KB 建设阶段 · 不覆盖运行时

## 6 · 总结 · perf-kp-sql 分工原则一句话

> perf-kp-sql 给 LLM 的 16 个主流程任务符合 P1-P4 · 其中 **2 个完全自动化(P1)**· **5 个有硬证据兜底(P2)**· **3 个 P1+P2 复合**· **2 个 P3+P2 复合(LLM 评 LLM 但有 P2 硬证据)**· **3 个是 LLM 唯一选项有降级路径(P4)**· **1 个待补 P2(5.3 一句话摘要 · 0.48.0 起新增)**。节后规划 3 个 P3 介入点(2.1.1 / 2.1.2 / 2.2.1)在 `2026-05-post-holiday-plan.md` 已排入跨家评审 + 5% 人工抽样校准。
>
> **没有"反正它出来了"的环节**:每个 LLM 决策点都能逐条说出"这一步为什么必须 LLM / 失败模式是什么 / 我们怎么知道它没乱编"。
>
> 仍存在 3 个**可解释性风险点**(主要问题摘要 / Phase 6 追问 / 根因选择推理链)· 已列改进路线 · 工程量 0.5-3 天。
